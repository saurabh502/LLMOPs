Introduction to Version Control

What is Version Control?

Version Control Systems (VCS) are tools that help track changes to files over time. Think of it like an "undo" button for your entire project, but much more powerful. It allows you to:

  • Track every change made to your codebase
  • Revert to previous versions if something breaks
  • Collaborate effectively with multiple developers on the same project without overwriting each other's work
  • Understand the history of a project â€“ who changed what, when, and why

Why Git?

Git is the most popular distributed version control system (DVCS) today. It's fast, flexible, and designed for collaboration. "Distributed" means that every developer has a full copy of the project history on their local machine, making most operations very fast and allowing offline work.

Git Basics and Workflows

Core Concepts:

Repository (Repo)

A directory containing your project files and a hidden .git subdirectory where Git stores all the history and metadata.

Commit

A snapshot of your project at a specific point in time. Each commit has a unique ID (a hash) and a commit message describing the changes.

Working Directory

The actual files and folders you see in your project directory.

Staging Area (Index)

An intermediate area where you prepare changes before committing them. You select which modifications from your working directory you want to include in the next commit.

Branch

An independent line of development. The default branch is usually called main or master. Branches allow you to work on new features or bug fixes without affecting the main codebase.

Remote

A version of your repository hosted on a server (like GitHub, GitLab, Bitbucket). This allows collaboration and backup. origin is the default name for the remote you cloned from.

Essential Commands:

Command Description
git init Initializes a new, empty Git repository in the current directory
git clone <repository_url> Creates a local copy of a remote repository
git status Shows the current state of your working directory and staging area
git add <file_name> or git add . Adds changes from your working directory to the staging area
git commit -m "message" Creates a new commit with the staged changes
git log Shows the history of commits
git branch Lists all local branches
git checkout <branch_name> Switches your working directory to the specified branch
git merge <branch_name> Merges the specified branch's history into your current branch
git pull Fetches changes from the remote repository and merges them
git push Sends your committed local changes to the remote repository

Basic Workflow Example:

# Clone (if starting from existing remote repo)
git clone <url>

# Or Initialize (if starting new local repo)
git init

# Create a branch for your work
git checkout -b my-new-feature # creates and switches

# Make changes: Edit your project files

# Check status
git status # see what you changed

# Stage changes
git add <file1> <file2> or git add .

# Commit changes
git commit -m "Implement user login feature"

# Push your branch (first time)
git push -u origin my-new-feature # -u sets the upstream tracking

# Push subsequent changes
git push

# Later, merge: After review (e.g., via a Pull Request on GitHub), merge the feature branch into main

Branching Strategies

Branching allows teams to work on different things in parallel without interference. Choosing a strategy depends on team size, project complexity, and release cycle.

Gitflow

Branches:

  • main (production-ready code)
  • develop (integration branch)
  • feature/* (new features)
  • release/* (preparing releases)
  • hotfix/* (critical fixes)

Pros:

Very structured, good for projects with scheduled releases. Clear separation of concerns.

Cons:

Can be complex, many long-lived branches.

GitHub Flow

Branches:

  • main (always deployable)
  • feature/* (branched off main)

Pros:

Simple, fast, good for continuous deployment/integration (CI/CD).

Cons:

Requires robust testing before merging to main. main must always be stable.

GitLab Flow

Branches:

  • main (production)
  • feature/* (branched off main)
  • Environment branches (e.g., staging)

Pros:

Balances simplicity with structure for multiple environments.

Cons:

Can add slight overhead compared to GitHub Flow.

Which to Choose?

  • Small teams, web apps, CI/CD: GitHub Flow is often preferred.
  • Larger teams, complex projects, scheduled releases: Gitflow might be suitable.
  • Need for staging/multiple environments: GitLab Flow provides a good model.

Managing Sensitive Information

WARNING

NEVER commit sensitive data directly into your Git repository! This includes API keys, passwords, database credentials, and private certificates.

Why?

  • Git history is generally permanent. Even if you remove the secret in a later commit, it still exists in the history.
  • Pushing to public repositories (like GitHub) exposes your secrets to everyone.
  • Even in private repositories, it increases the risk of exposure if the repo is compromised or access control is misconfigured.

How to Handle Secrets

.gitignore

Create a file named .gitignore in your repository's root directory. List files, directories, or patterns that Git should ignore (never track).

# Configuration files
config.prod.json
.env

# Log files
*.log
logs/

# Dependency directories
node_modules/
vendor/

# Build output
dist/
build/

# OS generated files
.DS_Store
Thumbs.db

Add .gitignore itself to the repository:

git add .gitignore && git commit -m "Add .gitignore"

Environment Variables

Store secrets in environment variables on the server or development machine where the code runs. Your application reads these variables at runtime. This is a very common and secure practice.

Configuration Files (Outside Repo or Templates)

Keep actual configuration files with secrets outside the Git repository. Commit a template or example configuration file (e.g., config.example.json) to the repository, showing the structure but using placeholder values. Developers or deployment scripts copy the template and fill in the real secrets on the target machine.

Secret Management Tools

Use dedicated tools like HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, Google Secret Manager, or Doppler. These tools securely store and manage access to secrets. Your application fetches secrets from these services at runtime.

Removing Accidentally Committed Secrets

This is difficult and dangerous because it involves rewriting repository history. If you push sensitive data:

  1. Immediately revoke/change the secret! Assume it's compromised.
  2. Clean the repository history using tools like BFG Repo-Cleaner (recommended) or git filter-branch.

WARNING: Rewriting history is a destructive operation. It changes commit IDs and requires all collaborators to re-clone or perform complex git pull --rebase operations. Coordinate carefully!

Conclusion

Version control with Git is an essential skill for modern software development. Understanding the basics, adopting a suitable branching workflow, and diligently managing sensitive information will significantly improve your productivity, collaboration, and project security. Start simple, practice the commands, and gradually incorporate more advanced techniques as needed.
